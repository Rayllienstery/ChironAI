# Indexing Content

RAG is only as good as the chunks in Qdrant. The **Crawler / Indexer** tab and host ingestion pipelines turn source documents into embedded points with metadata. This guide covers the operational path from raw docs to a build-ready collection.

## End-to-end pipeline

```
Sources (git, HTTP, files) → Crawler / pipeline → Chunk + embed → Upsert Qdrant
     → Verify in RAG tab → Attach collection to build → Test via Model Tester
```

Plan for iteration: first index a **small** representative sample, validate retrieval, then scale up.

<a id="sources"></a>

## Sources

Deployment-specific, but commonly:

- Local markdown / code trees (project docs, ADRs)
- Git repositories (branch-filtered)
- Crawled HTTP sites (respect robots and rate limits)
- Preprocessed exports from CI

In **Crawler / Indexer → Add Source**, configure:

| Field | Purpose |
|-------|---------|
| **Source ID** | Stable name referenced when creating collections |
| **URL** | Root documentation or site entry point |
| **Max depth** | How many link hops Playwright follows (1–5) |
| **Crawler** | Engine (Playwright) |
| **Doc only** | Restrict crawl to documentation-like pages |
| **Seed URLs** | Extra entry points beyond the root URL |

<a id="seed-urls"></a>

### Seed URLs and doc_only

Seed URLs help when documentation is split across paths that the root URL does not discover. **Doc only** filters navigation-heavy pages before markdown is stored — useful for vendor doc sites.

<a id="crawl-actions"></a>

## Crawl actions

From the Crawler section header:

| Action | When to use |
|--------|-------------|
| **Crawl** | Refresh the currently highlighted source row |
| **Crawl selected** | Batch refresh for checkbox-selected sources |
| **Crawl ALL** | Re-crawl every configured source |

Watch **Indexed** counts in the sources table. Expand a row to inspect recent pages and chunk counts before creating a collection.

<a id="md-pipeline"></a>

## MD Pipeline (optional)

Before chunking, apply a named **MD Pipeline** to strip boilerplate, remove nav soup, normalize whitespace, or reject low-signal bodies. Switch to the **MD Pipeline** section tab to:

1. Select or create a pipeline
2. Add ordered steps (see step types below)
3. **Preview on file** against a crawled markdown sample
4. **Save** the pipeline for indexing runs

<a id="md-step-types"></a>

### MD step types

Common step types (full list in the UI **Add step** menu):

| Type | Typical use |
|------|-------------|
| `strip_meta_block` | Parse leading `<!-- meta ... -->` and keep body only |
| `delete_lines_exact` | Drop boilerplate lines (e.g. "Table of Contents") |
| `delete_lines_containing` | Remove lines with substrings (locale switchers) |
| `strip_sections_by_heading` | Remove autogenerated sections by heading name |
| `normalize_whitespace` | Collapse extra spaces outside code fences |
| `reject_low_signal_body` | Drop pages that are mostly links / too short |

<a id="create-collection"></a>

## Create Collection

**Create Collection** opens the indexing wizard:

1. **Collection name** — becomes the Qdrant collection id (use a domain-specific name)
2. **Select sources** — only crawled sources with pages
3. **Embedding** — provider + model (same pool as RAG tab)
4. **Chunking** — max/min size, confidence, top_k
5. Monitor live progress; cancel if the embed provider is overloaded

<a id="embedding"></a>

### Embedding provider and model

All points in a collection must use the **same embedding model** as queries at retrieval time. The Create Collection modal shares the RAG embedding catalog.

<a id="parallel"></a>

#### Parallel embedding workers

**Parallel embedding requests** (1–8) trade speed against GPU/VRAM pressure. Lower this value if you see provider timeouts during large indexes.

<a id="chunking"></a>

### Chunking parameters

| Knob | Effect |
|------|--------|
| Chunk max / min | Larger chunks = more context per hit, less precise matching |
| Confidence threshold | Filters weak chunks during indexing (pipeline-dependent) |
| Top K | Default retrieval breadth hint stored with collection metadata |

When chunks lack metadata, intent filters (symbol/framework) cannot narrow results — retrieval still works but may feel “fuzzy”.

## Typical operator workflow

1. **Add Source** — URL, depth, seed URLs, doc_only ([sources](#sources))
2. **Crawl** — single, selected, or ALL ([crawl actions](#crawl-actions))
3. **(optional) MD Pipeline** — clean markdown ([md-pipeline](#md-pipeline))
4. **Create Collection** — embed + chunk → Qdrant ([create-collection](#create-collection))
5. **Verify in RAG tab** — collection visible, retrieval test
6. **Attach to Build** — `rag_collection` in LLM Proxy wizard ([LLM Proxy Builds](builds))

## Embedding model consistency

If you change embed models in **RAG Fusion Proxy → Model settings**:

1. Plan downtime or new collection name
2. Re-index entire corpus
3. Update builds to point at new collection
4. Retire old collection after verification

Mixing old and new vectors in one collection produces nonsense similarity scores.

## Re-indexing and updates

| Event | Action |
|-------|--------|
| Doc content changed | Incremental or full reindex (pipeline-dependent) |
| New major OS/SDK release | New collection or version suffix (`docs-ios-19`) |
| Bad retrieval after config tweak | Re-run retrieval tests; consider re-chunk |
| Qdrant reset / new volume | Full reindex mandatory |

Keep a runbook note: last successful index time, source commit hash, collection name, embed model id.

## Crawler tab tips

- Start with a small source or shallow max depth when experimenting
- Watch disk: Qdrant storage grows with chunk count and vector dimension
- Parallel jobs may contend for GPU/CPU used by embedding — stagger heavy jobs
- Network crawls: configure timeouts; hung jobs block operator visibility

<a id="troubleshooting-crawler"></a>

## Troubleshooting crawler / indexer

| Symptom | Check |
|---------|-------|
| Empty sources table | Add a source; confirm crawler service permissions |
| Crawl stuck at 0 indexed | VPN/firewall, Playwright deps, URL reachability |
| Create Collection fails | Embed provider online; sources have pages; see **Logs** |
| Collection in UI but 0 points | Job cancelled or embed model mismatch |
| Good crawl, bad retrieval | Chunk sizes, MD pipeline too aggressive, wrong collection on build |

## Permissions and paths

Common failures:

- WebUI service account cannot read UNC/share paths
- Docker crawler container lacks volume mount for host docs
- Git credentials missing for private repos
- SSL errors on internal HTTPS doc servers

Fix permissions at the OS/container level — the UI only reports the failure.

## Quality checklist before production

- [ ] Retrieval test returns expected doc titles for 5+ known queries
- [ ] Model Tester shows relevant context snippets (not empty / unrelated)
- [ ] Collection point count stable after job completes
- [ ] Builds reference correct collection name
- [ ] RAG Tests pass (if used in CI)
- [ ] Disk backup strategy for Qdrant volume documented
